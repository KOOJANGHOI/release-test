## 📢 [공지] 릴리스 브랜치 생성 및 PR 병합 자동화 스크립트 배포

안녕하세요 개발팀 여러분! 🚀

매 배포마다 반복되던 **'릴리스 브랜치 생성, ready-to-merge PR 일일이 확인 후 병합, 릴리스 PR 생성'** 과정을 한 방에 처리해 주는 자동화 스크립트가 완성되어 공유해 드립니다.

과거 정방향(`--ff-only`) 병합 시 줄기가 꼬여 발생하던 `fatal: 정방향이 불가능하므로, 중지합니다.` 에러를 완벽히 해결하고, 깔끔한 커밋 히스토리를 유지할 수 있도록 **Squash Merge 기반**으로 설계되었습니다.

---

### 🎯 스크립트 구성 (2개 제공)

프로젝트 특성에 맞춰 **서버 배포용**과 **데스크탑 배포용** 2가지 스크립트를 제공합니다:

1. **`create-release-server.sh`** - 서버 배포 전용
   - 릴리스 브랜치 형식: `release/YYYYMMDD` (예: `release/20260619`)
   - 배포일 입력: YYYYMMDD 형식 (예: 20260619)
   - 모든 `ready-to-merge` 라벨 PR 대상

2. **`create-release-desktop.sh`** - 데스크탑 배포 전용
   - 릴리스 브랜치 형식: `release/desktop/{version}` (예: `release/desktop/1.0.83`)
   - 버전 입력: x.y.z 형식 (예: 1.0.83)
   - `desktop/`으로 시작하는 브랜치의 `ready-to-merge` PR만 대상

---

### 🛠️ 주요 자동화 기능 (Workflow)

1. **사전 검증:** 로컬 작업 트리(`git status`) 청결 여부 및 GitHub CLI(`gh`) 인증 상태 자동 확인

2. **최신화:** `develop` 브랜치 원격 기준 강제 최신화

3. **릴리스 브랜치 처리:**
   - 새로운 릴리스: 배포일/버전 기준 릴리스 브랜치 자동 생성
   - 기존 릴리스: 동일한 배포일/버전의 릴리스 브랜치가 이미 존재하면 해당 브랜치를 체크아웃하여 이어서 진행

4. **스마트 타겟팅:** GitHub에서 **`ready-to-merge`** 라벨이 붙은 대기 중인 오픈 PR 목록을 자동으로 조회
   - 서버: 모든 ready-to-merge PR
   - 데스크탑: `desktop/`으로 시작하는 브랜치의 ready-to-merge PR만 필터링

5. **인터랙티브 병합:** 조회된 PR을 하나씩 보여주며 배포 포함 여부(`y/n`) 확인 후, 선택된 PR들을 **Squash 병합**

6. **표준 커밋 메시지 준수:** 깃허브 웹에서 수동으로 스쿼시 머지했을 때와 동일한 포맷인 **`PR제목 (#PR번호)`** 형태로 커밋 메시지 자동 생성

7. **PR 관리 및 알림:** 병합 성공 시 해당 깃허브 PR을 즉시 `Close`하고, 배포본에 포함되었다는 안내 댓글을 자동으로 작성

8. **최종 릴리스 PR 생성:** 병합된 릴리스 브랜치를 원격에 푸시한 후, 릴리스 브랜치 ➔ `develop`으로 가는 최종 릴리스 PR까지 자동 생성

9. **충돌 감지 및 리포팅:** Merge Conflict 발생 시 자동으로 병합을 중단하고, 최종 결과 리포트에서 성공/실패/스킵된 PR 목록을 한눈에 확인 가능

### 🧪 안전한 테스트를 위한 `--dry-run` (모의 실행) 옵션 지원

실제 깃허브 원격 저장소에 영향을 주지 않고, **로컬에서 병합 충돌이나 시나리오를 미리 검증해 볼 수 있는 모의 실행 모드**를 지원합니다.

```bash
# 1. 서버 배포 - 모의 실행 (원격 push X, 깃허브 PR 조작 X, 테스트 완료 후 로컬 브랜치 자동 삭제)
./create-release-server.sh --dry-run

# 2. 서버 배포 - 실제 운영 실행
./create-release-server.sh

# 3. 데스크탑 배포 - 모의 실행
./create-release-desktop.sh --dry-run

# 4. 데스크탑 배포 - 실제 운영 실행
./create-release-desktop.sh
```

> **💡 Tip:** 스크립트 내부적으로 `set -euo pipefail` 안전장치가 적용되어 있어, 중간에 예기치 못한 에러나 오타 변수가 감지되면 스크립트가 혼자 폭주하지 않고 그 즉시 안전하게 실행을 중단합니다.

---

### 📦 실행 요구사항 및 사용 방법

- **필수 도구:** `git` 및 GitHub CLI (`gh`) 설치 및 로그인 완료 상태 (`gh auth login`)
- **스크립트 위치:** 프로젝트 루트 폴더
  - `./create-release-server.sh` (서버 배포용)
  - `./create-release-desktop.sh` (데스크탑 배포용)
- **권한 부여:** 실행 전 실행 권한이 없다면 다음 명령어를 실행해 주세요:
  ```bash
  chmod +x ./create-release-server.sh
  chmod +x ./create-release-desktop.sh
  ```

### 🎬 실행 예시

**서버 배포 (날짜 기반):**
```bash
$ ./create-release-server.sh
배포일을 입력하세요 (YYYYMMDD): 20260619
Release Branch : release/20260619
...
```

**데스크탑 배포 (버전 기반):**
```bash
$ ./create-release-desktop.sh
배포 버전을 입력하세요 (예: 1.0.83): 1.0.83
Release Branch : release/desktop/1.0.83
...
```

### ✨ 이어서 진행 기능 (Resumable Release)

이미 진행 중인 릴리스에 추가로 PR을 병합하고 싶을 때, **같은 배포일/버전을 다시 입력**하면 기존 릴리스 브랜치를 자동으로 인식하여 이어서 작업할 수 있습니다.

예시:
1. 첫 번째 실행: `release/desktop/1.0.83` 생성 후 PR 3개 병합
2. 추가 PR이 ready-to-merge로 올라옴
3. 두 번째 실행: 다시 버전 `1.0.83` 입력 → 기존 브랜치에 추가로 병합
